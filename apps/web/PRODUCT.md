# Product

## Register

product

## Users

Code Cat is used by three groups inside a classroom learning loop. Students use it on laptops or school lab machines to play short programming rooms, track their progress, and complete classroom assignments. Teachers use it to create classrooms, enroll students, build custom rooms, assign gameplay, and review outcomes without digging through technical data. Admins use it to monitor adoption, content rollout, and overall system health.

## Product Purpose

Code Cat turns introductory coding lessons into playable puzzle rooms while keeping classroom management practical for teachers. It exists to make programming concepts feel learnable through action, then connect that gameplay back to real classroom structure through enrollment, assignments, progress, and review. Success looks like a student immediately understanding what to play next, and a teacher being able to create a classroom, attach gameplay, and inspect progress without confusion or UI friction.

## Brand Personality

Focused, playful, trustworthy. The product should feel like a serious classroom tool with game energy, not a toy and not a sterile dashboard. It should feel structured enough for teachers to trust during a capstone demo, while still giving students a sense of momentum and personality.

## Anti-references

- Generic SaaS dashboards with thin hierarchy, gray-on-gray controls, and interchangeable cards.
- Overly kiddie ed-tech interfaces that feel juvenile or cluttered.
- Neon gamer UI with excessive glow, chrome, and noisy decoration that fights readability.
- Decorative dark-mode interfaces where every panel has a border, glow, gradient, and shadow at once.

## Design Principles

- Put the classroom task first. The interface should always make the next action obvious for the current role.
- Separate progress contexts clearly. Built-in progression and classroom assignments must never feel mixed together.
- Preserve playfulness in controlled moments. Branding and game tone should show up in key moments, not overwhelm the workflow.
- Optimize for scanning before exploration. Teachers and students should understand status, counts, and next steps at a glance.
- Reduce UI dialects. The system should feel like one product, not a stack of unrelated panels and page styles.

## Accessibility & Inclusion

Target WCAG AA contrast across the product, including placeholder text and status chips. Respect reduced motion by keeping animation subtle and non-essential. Prioritize readable type, clear focus states, and classroom-friendly contrast that remains legible on lower-quality displays and projected screens.
